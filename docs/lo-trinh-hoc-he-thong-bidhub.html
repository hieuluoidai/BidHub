Lộ trình học đầy đủ để hiểu và bảo vệ hệ thống BidHub

Mục tiêu của tài liệu này là giúp người chưa nắm chắc code vẫn có thể học theo thứ tự đúng, hiểu hệ thống theo luồng xử lý thực tế, và chuẩn bị tốt cho phần thuyết trình hoặc vấn đáp với thầy cô.

Tóm tắt hệ thống: BidHub là một hệ thống đấu giá trực tuyến gồm client JavaFX, server Java socket, MySQL, và một HTTP image server để phục vụ upload / tải ảnh.

Thứ tự học khuyến nghị: kiến trúc tổng thể - luồng client - luồng server - nghiệp vụ đấu giá - dữ liệu / database - vận hành production - phần tự học ngoài slide.

2. Lộ trình học theo 8 buổi

Buổi 1 - Dựng bản đồ tổng thể của hệ thống 2 giờ

Mục tiêu

  • Nói được hệ thống gồm những thành phần nào.
  • Biết client, server, database, image server liên hệ với nhau ra sao.
  • Biết vai trò của ba loại người dùng: Bidder, Seller, Admin.

File cần đọc

  • README.md
  • UML.md
  • pom.xml
  • src/main/resources/server.properties

Việc phải hiểu

  • Client JavaFX chạy trên máy người dùng.
  • Server Java nhận request đấu giá qua socket TCP cổng 1234.
  • Ảnh đi qua HTTP cổng 8080, không đi qua socket đấu giá.
  • MySQL lưu toàn bộ dữ liệu bền vững như user, auction, bid history, notification.

Câu hỏi tự kiểm tra

  1. Tại sao hệ thống này không thể chỉ là một app local chạy một máy?
  2. Vì sao cần tách socket đấu giá và HTTP ảnh thành hai kênh khác nhau?
  3. Nếu server chết thì những phần nào của hệ thống dừng ngay?

Dựa trên phần đã học OOP, kiến trúc Java, design overview.

  • Hiểu AppState giữ state toàn cục gì và vì sao cần nó.
  • Hiểu client nhận dữ liệu từ server rồi phân phối về UI bằng listener.

Điểm dễ bị hỏi

Tự học thêm JavaFX MVC, synchronization giữa UI thread và thread mạng không phải phần được dạy đậm trên lớp.

Buổi 4 - Học luồng nghiệp vụ cốt lõi: đăng nhập, tạo phiên, đặt giá 3 giờ

Nhiệm vụ

  • Tự kể lại bằng miệng 3 luồng: đăng nhập, tạo phiên đấu giá, đặt giá thủ công.
  • Mỗi luồng phải nói được: ai gọi ai, dữ liệu đi qua lớp nào, lớp nào là nơi ra quyết định.
  • Đặc biệt với luồng đăng nhập, phải nói đúng bản mới: LoginController -> AuctionClient.authenticate -> ClientHandler -> UserHandler -> AuthService -> UserDAO -> MySQL.

  • src/main/java/network/AuctionClient.java
  • src/main/java/model/auth/AuthRequest.java
  • src/main/java/model/auth/AuthResponse.java
  • src/main/java/network/handler/UserHandler.java
  • src/main/java/service/AuthService.java
  • src/main/java/service/AuctionService.java
  • src/main/java/controller/CreateSessionController.java
  • src/main/java/controller/BidController.java

Bài tập bắt buộc

  1. Vẽ lại sequence đăng nhập bằng tay, không nhìn UML.
  2. Vẽ lại sequence tạo phiên đấu giá từ UI đến DB.
  3. Vẽ lại sequence đặt giá thành công và đặt giá thất bại.

Cần nói thêm khi vấn đáp Hệ thống đã bắt đầu chuyển đúng sang mô hình client-server ở phần auth, nhưng chưa chuyển hết mọi màn hình; vẫn còn một số chỗ client chạm DB trực tiếp.

Dựa trên phần đã học Đây là nơi bạn áp dụng OOP, MVC, exception handling, service layer.

Buổi 8 - Học phần triển khai thực tế và chuẩn bị vấn đáp 3 giờ

Mục tiêu

  • Biết hệ thống đã được deploy thật trên VPS như thế nào.
  • Biết những tối ưu production đã làm gần đây.
  • Chuẩn bị câu trả lời cho câu hỏi “bạn tự làm gì, phần nào là tự học ngoài lớp”.

Nội dung phải nắm

  • Server chạy dưới systemd, không còn nohup thủ công.
  • Đã thêm swap để VPS 1 GB RAM ổn định hơn.
  • Đã siết UFW, bỏ public MySQL, khóa MySQL về 127.0.0.1.
  • Đã thêm cơ chế bỏ qua traffic không đúng protocol để giảm log rác.
  • Đã vá runtime schema drift cho notifications.source_id.

Cách nói nếu bị hỏi sâu

  • “Phần này không phải trọng tâm slide lớp, em tự nghiên cứu thêm khi triển khai thật.”
  • “Em có thể giải thích theo mức kiến trúc và mục tiêu kỹ thuật, không đi sâu đến mức quản trị hệ thống chuyên nghiệp.”

Tự học thêm Đây là phần production thực tế, gần như chắc chắn vượt ra ngoài nội dung lớp.
